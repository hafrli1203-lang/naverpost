# 블라이 문서 기반 구현 계획

## 목적

이 문서는 `C:\Users\hafrl\Desktop\블로그 블라이 기능.docx` 내용을 현재 프로젝트에 반영하기 위한 구현 계획서다.

가장 중요한 원칙은 다음과 같다.

- 기존 프로세스는 변경하지 않는다.
- 현재 UX는 변경하지 않는다.
- 현재 정보성 중심 글쓰기 프롬프트의 뼈대는 유지한다.
- 바꾸는 것은 추천 정밀도, 내부 브리프, 검수 정밀도, 중복 회피 방식이다.
- 추측성 규칙이나 임의 점수식으로 설계하지 않는다.
- 블라이 문서에 나온 실제 판단 기준과 네이버에서 확인 가능한 신호만 사용한다.

## 유지할 기존 프로세스

아래 파이프라인은 그대로 유지한다.

1. 시작 단계에서 매장 / 카테고리 / 홍보성·정보성 / 말투 / 글자수 결정
2. 제목/키워드 추천
3. 선택된 키워드로 Perplexity 자료 수집
4. 글 작성
5. 내부 검수/보정
6. 이미지 생성

즉 사용자는 계속 지금처럼 다음만 선택한다.

- 매장 선택
- 카테고리 선택
- 제목/키워드 선택

## 문서에서 확인한 실제 반영 기준

### 1. 형태소 분석

문서 기준으로 반영해야 하는 항목:

- 제목 형태소 분석
- 본문 형태소 반복수
- 본문 형태소 종류 수
- 목표 키워드와 본문 형태소의 주제/테마 정합성

의미:

- 반복수가 높은 형태소는 엔진의 주목도가 높아질 수 있다.
- 제목은 형태소 단위로 해석되므로 의도와 다른 의미의 형태소가 들어가면 안 된다.
- 같은 길이의 글이라도 형태소 종류 수가 적으면 정보량이 부족할 수 있다.

### 2. 비속어/상업어 분석

문서 기준으로 반영해야 하는 항목:

- 비속어
- 욕설
- 성인어
- 상업어
- 강조어
- 광고성 표현

의미:

- 비속어, 욕설, 성인어는 제거 대상이다.
- 상업어, 강조어, 광고 표현은 과도 사용 시 위험 신호다.

### 3. 제목-본문 구조 분석

문서 기준으로 반영해야 하는 항목:

- 제목에 포함된 키워드 색인 요소
- 본문에서 해당 색인이 실제 활성화되는지 여부
- 제목과 본문 간 유사성
- 제목과 본문 간 일치율

의미:

- 제목 키워드가 본문에서도 자연스럽게 살아야 한다.
- 제목만 세고 본문 연결이 약하면 일치율이 떨어진다.

### 4. 노출 관련 실제 신호

문서 기준으로 반영해야 하는 항목:

- 유사도
- 최신성
- 근접도
- 적합도
- 블로그 품질 지수

의미:

- 유사도는 검색 키워드와 글의 유사성이다.
- 최신성은 최근 글 우대와 연결된다.
- 근접도는 질의 원형 포함과 관련 있다.
- 적합도는 여러 요소가 합쳐진 종합 랭킹 신호다.

### 5. 글 품질에 영향을 주는 관찰값

문서 기준으로 확인된 항목:

- 글자수
- 이미지 수
- 스크랩

문서에는 블라이 관찰값으로 다음 설명이 있다.

- 글 품질은 블로그 등급 + 글자수 + 이미지수 + 스크랩에 영향을 받는다.
- 단, 이 값은 네이버 내부 관측 지표를 기반으로 한 설명이므로 현재 저장소만으로 전부 계산할 수는 없다.

### 6. 본문 내 실제 반영 요소

문서 기준으로 순위 영향이 있다고 본 항목:

- 이미지 캡션 텍스트
- 글감 안 텍스트
- 표 안 텍스트
- 인용구 안 텍스트
- 첨부파일 파일명

문서 기준으로 핵심 영향 요소로 보지 않는 항목:

- 이미지 파일명
- 동영상 제목
- 동영상 태그
- 링크 안 텍스트
- 지도 안 텍스트
- 글 태그

## 구현 원칙

### 1. 제목/키워드 추천기는 유지하되 후처리를 강화한다

현재 `keywords` 단계는 유지한다.

바꿀 점:

- 추천 결과를 그대로 반환하지 않는다.
- 문서 기준 검사 메타를 붙인다.
- 같은 매장 중복 위험을 계산한다.
- 6개 블로그 내부 중복 위험을 계산한다.
- 제목 형태소 정합성과 상업어 위험을 검사한다.
- 본문으로 자연스럽게 확장 가능한지 내부 판정을 붙인다.

중요:

- 추천기를 새 UX로 바꾸지 않는다.
- 서버 내부 후처리와 정렬만 강화한다.

### 2. 글쓰기 전에 내부 브리프를 추가한다

사용자에게 보이는 새 단계는 추가하지 않는다.

내부 브리프 구성:

- 선택된 제목
- 메인/서브 키워드
- 글 유형
- 말투
- 글자수
- 매장/카테고리 정보
- Perplexity 자료 요약
- 해당 블로그의 기존 글 이력
- 다른 블로그들의 같은 매장 방향
- 피해야 할 중복 패턴
- 제목 형태소를 본문에서 자연스럽게 살리는 규칙

의도:

- 기존 프롬프트를 교체하는 것이 아니라,
- 기존 프롬프트가 더 정확히 작동하도록 입력 품질을 높인다.

### 3. 검수는 문서 기준으로 확장한다

현재 단순 검수 항목만으로는 부족하다.

추가할 내부 분석 항목:

- 형태소 반복수
- 형태소 다양성
- 제목 형태소 활성화
- 제목-본문 일치율
- 상업어/강조어 과다 사용
- 표/구조 요소 반영 상태
- 같은 매장 구조 중복
- 네트워크 내부 제목/키워드 중복

중요:

- 내부 점수 표현은 사용할 수 있다.
- 그러나 계산식은 문서 체크 항목 기반이어야 한다.
- 임의 가중치를 먼저 만드는 방식은 금지한다.

### 4. 자동 보정은 1회만 허용한다

현재 `src/app/api/article/route.ts`의 자동 보정은 최대 2회다.

변경 원칙:

- 최대 1회로 제한
- 보정도 문체 재작성보다 문서 기준 결함 보완 중심

이유:

- 반복 보정은 속도를 해친다.
- 문체가 무너질 수 있다.
- 기존 프롬프트 전략을 훼손할 수 있다.

## 단계별 개발 순서

### 1단계. 문서 기준 타입 정의

목표:

- 문서에서 확인한 판별 항목을 코드 타입으로 먼저 고정

작업:

- `src/types/index.ts` 확장
- 형태소 분석 결과 타입 정의
- 제목-본문 정합성 결과 타입 정의
- 중복 분석 결과 타입 정의
- 내부 브리프 타입 정의

### 2단계. 형태소/언어 분석기 추가

목표:

- 제목과 본문을 문서 기준으로 분석 가능하게 만들기

작업:

- 제목 형태소 추출
- 본문 형태소 추출
- 반복수 계산
- 형태소 종류 수 계산
- 상업어/강조어/광고성 표현 사전 정리

주의:

- 이 단계는 문서 기반 규칙 구현이다.
- 아직 네이버 실데이터 연동 단계가 아니다.

### 3단계. 제목/키워드 추천 후처리 강화

목표:

- 추천 품질을 내부적으로만 높이고 UX는 유지

작업:

- `keywords` API 응답에 분석 메타 추가
- 추천 결과 정렬 기준 강화
- 같은 매장과 네트워크 중복 위험 비교
- 제목 형태소 위험 및 상업어 위험 표시

### 4단계. 글쓰기 전 내부 브리프 생성

목표:

- 기존 article/promo prompt 앞단에 정리층 추가

작업:

- `buildArticleBrief()` 추가
- Perplexity 결과를 브리프용 요약으로 정리
- 제목-본문 연결 규칙 포함
- 중복 회피 방향 포함

### 5단계. 본문 검수기 교체

목표:

- 기존 금칙어 중심 검수에서 문서 기준 검수로 확장

작업:

- `contentValidator.ts` 확장 또는 분리
- 형태소 분석 결과 반영
- 제목-본문 구조 분석 반영
- 구조 요소 반영 상태 분석
- 중복 구조 분석 반영

### 6단계. 자동 보정 1회 제한

목표:

- 검수 미달 시 1회만 보정

작업:

- `MAX_REVISION_ATTEMPTS = 1`
- `revisionPrompt`를 문서 기준 결함 수정형으로 개편

### 7단계. 네이버 실데이터 연동 설계

목표:

- 현재 저장소 밖의 실제 검색/노출 데이터를 붙일 수 있도록 인터페이스 준비

작업:

- 검색 결과 수집 포인트 정의
- 자동완성/연관검색어/트렌드 데이터 소스 정의
- 블로그 품질 관련 외부 입력 구조 정의

주의:

- 이 단계는 설계 우선
- 구현은 실제 접근 가능한 데이터 소스 확인 후 진행

- 자격 증명이 없거나 호출 실패 시 `status: "unavailable"`로 명시
- 자동완성 / 스마트블록 / 인기글 / 통합검색 파서는 아직 미구현
- 추측값은 넣지 않음

### 아직 미완료인 항목

아래 항목은 실제 외부 데이터 접근 또는 추가 구현이 필요하다.

1. 네이버 검색 실데이터 파서

- 실제 검색 결과 페이지 기준 순위
- 노출 영역별 분류
- 스마트블록 / 인기글 / 블탭 / 인플탭 판별

2. 자동완성 / 연관검색어 / 검색량 트렌드 연결

- 현재는 인터페이스만 존재
- 실제 데이터 소스 연결 필요

3. 블로그 품질 관련 외부 운영 데이터 연계

- 스크랩 수
- 발행 후 노출률
- 누락 현황
- 실제 블로그 품질 관측값

4. 실제 형태소 엔진 교체

- 현재 표면 토큰 분석을 실제 한국어 형태소 분석기로 교체 필요

## 이어서 작업할 때 우선순위

1. 실데이터 수집 방식 확정
2. 네이버 검색 결과 / 자동완성 / 연관검색어 파서 연결
3. 외부 신호를 `externalSignals.status = "available"`로 전환
4. 필요 시 실제 형태소 엔진 도입 검토

## 세션 복구 메모

이 문서만 다시 읽어도 다음 사실을 전제로 이어가면 된다.

- 기존 프로세스는 유지한다.
- 현재 구현은 추천기 / 내부 브리프 / 검수기 / 보정 1회 / 외부 연동 틀까지 반영되었다.
- 아직 실제 네이버 검색 데이터는 연결되지 않았다.
- 외부 데이터가 없는 상태에서 추측값을 추가하면 안 된다.
