# v1.6 — 키워드 맥락 조사 강화 + 채팅 기반 재수정

## 배경 / 문제

생성된 본문이 키워드의 실제 업계 의미와 어긋나는 사례가 발생.
- "기능성렌즈"(현장: 눈 피로 감소 안경렌즈)를 "코팅 기능"으로 서술
- "멀티포컬"(콘택트렌즈)을 "다초점 안경렌즈"로 서술

### 근본 원인 (코드 추적 결과)
1. **조사 타임아웃 회귀**: `cb4cb18`(2026-05-07)에서 `RESEARCH_TIMEOUT_MS` 45s → 12s.
   Perplexity sonar가 12초를 자주 초과 → 조사 자료가 빈 값으로 폴백 →
   Claude가 일반 지식만으로 작성 → 일반적 오해가 본문에 반영.
2. **맥락 없는 검색**: `researchKeyword(keyword.mainKeyword)` — 메인 키워드 문자열만 전달.
   서브 키워드 2개와 카테고리(콘택트렌즈/안경렌즈)가 검색어에 포함되지 않음.
3. **얕은 단발 검색**: 후속 질문 5개를 응답으로 받기만 하고 재검색하지 않음.
4. **의미 검증 부재**: `validateContent`는 금지어·키워드 원형·글자수·표만 검사.
   주제-키워드 의미 정합성은 검증하지 않아 잘못된 글도 통과.
5. **사후 교정 수단 부재**: 지시 기반 재작성/채팅 기능 없음. (재작성=동일 파라미터 재생성)

## 목적

한 줄: 키워드를 정확한 업계 의미로 조사·작성하게 만들고, 어긋난 글을 사람이 채팅으로
지적해 재수정할 수 있게 한다.

## 범위 (확정된 결정)

- 조사 강화: 메인+서브2+카테고리 합쳐 1차 검색 + 후속 질문 2~3개 재검색으로 자료 보강
- 도메인 용어집(안경원 용어 사전) + 카테고리 맥락 주입 (Perplexity·본문 프롬프트 양쪽)
- 조사 타임아웃 복구 + 조사 실패 시 UI 폴백 표시
- 멀티턴 채팅 스레드 기반 재수정 기능

## STEP 1. 설계

### 1-A. 도메인 용어집

| 항목 | 정의 |
|------|------|
| 목적 | 모호한 업계 용어를 정확한 의미로 고정해 오해를 소스에서 차단 |
| 트리거 | 본문 생성·조사·채팅 재수정 시 키워드에 용어집 항목이 포함되면 자동 주입 |
| 실패 복구 | 용어집 파일 누락/파싱 실패 시 빈 배열 반환, 기존 흐름 유지 (조사·작성 차단 금지) |
| 상태 경계 | `lib/domain/opticalGlossary.ts` 소유. 순수 함수 + JSON 데이터 로드. 다른 모듈 미침범 |
| 아웃풋 | `lookupGlossary(keywords[]): GlossaryEntry[]`, `buildGlossaryHint(entries): string` |

데이터: `data/opticalGlossary.json` (사용자가 직접 항목 추가 가능)
```json
[
  { "term": "멀티포컬", "category": "콘택트렌즈",
    "means": "노안 교정용 콘택트렌즈", "notMeans": "다초점 안경렌즈가 아님" },
  { "term": "기능성렌즈", "category": "안경렌즈",
    "means": "눈 피로 감소 기능을 가진 안경렌즈", "notMeans": "단순 코팅 기능이 아님" }
]
```
매칭: 키워드 문자열에 `term`이 포함되면 매칭(공백 정규화, 부분일치).

### 1-B. Perplexity 조사 강화

| 항목 | 정의 |
|------|------|
| 목적 | 정확한 맥락으로 조사하고 후속 질문 재검색으로 자료를 풍부하게 |
| 트리거 | 본문 생성 시 1회 (route에서 호출) |
| 실패 복구 | 1차 검색 실패→빈 결과 폴백+상태 unavailable. 후속 검색 일부 실패→성공분만 병합 |
| 상태 경계 | `lib/ai/perplexity.ts`. 인터페이스 변경: `researchKeyword(params)` |
| 아웃풋 | `{ text, result, status: "ok"|"empty" }` |

흐름 (기본 원칙 — 사용자 확정):
1. 1차 검색어 = `메인 + 서브1 + 서브2` **3개 키워드 동시** + `(카테고리) + 용어집 의미 힌트` 결합 프롬프트
2. 응답에서 후속 질문 추출 → **5개 전부 병렬 재검색** (각 짧은 요약)
3. 1차 summary + citations + 후속 5개 답변 요약을 하나의 자료 텍스트로 병합
4. 전체에 타임아웃 예산 부여(아래)

### 1-C. 타임아웃 복구 + 폴백 표시

| 항목 | 정의 |
|------|------|
| 변경 | `RESEARCH_TIMEOUT_MS` 12s → 40s (멀티라운드 예산). 후속 검색은 내부 병렬+짧은 개별 한도 |
| 폴백 | 조사 status=empty면 `ArticleContent.researchStatus="empty"` 세팅 → UI 경고 배지 |
| 경쟁분석 | `COMPETITOR_ANALYSIS_TIMEOUT_MS` 12s → 25s |

### 1-D. 본문 프롬프트 용어 정의 주입

`buildArticlePrompt`/`buildPromoPrompt` [입력 정보] 아래에 `[키워드 정확한 의미]` 블록 추가.
용어집 매칭이 있으면 "X는 ~를 가리키며 ~가 아니다"를 명시. 카테고리도 함께 강조.

### 1-E. 멀티턴 채팅 재수정

| 항목 | 정의 |
|------|------|
| 목적 | 사람이 "왜 틀렸는지" 지적하면 그 지시대로 본문을 재수정 |
| 트리거 | 본문 미리보기(stage 2)에서 채팅 입력 전송 |
| 실패 복구 | API 오류 시 기존 본문 유지, 토스트 경고. 타임아웃 시 재시도 1회 |
| 상태 경계 | API `api/article/chat`, 프롬프트 `chatRevisionPrompt.ts`, UI `ArticleChat.tsx`. 대화는 workflow state(`article.revisionChat`)에 저장 |
| 아웃풋 | `{ content, validation, assistantNote }` → 본문 갱신 + 대화에 응답 추가 |

프롬프트 입력: 현재 본문 + 키워드/카테고리/용어집 + 대화 메시지 배열 + 보존 규칙
(키워드 원형 유지, 쉼표·이모지 금지, 매장 안내 보존, 글자수 유지).

타입 추가:
- `ChatMessage { role: "user" | "assistant"; content: string }`
- `ArticleContent.researchStatus?: "ok" | "empty"`
- `ArticleContent.revisionChat?: ChatMessage[]`
